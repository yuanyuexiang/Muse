# MUSE 架构设计

> **MUSE** — Menu Understanding & Service Engine
> 一个基于企业微信的 AI 会话处理平台，首个业务场景为餐饮设计公司的 AI 菜单设计与客户沟通。

本文档从**产品架构**视角设计，不绑定任何单一语言或框架立场。

---

## 一、项目定位

MUSE 的核心不是"菜单 AI"，而是一个通用的 **AI Conversation Platform（AI 会话处理平台）**。"菜单整理"只是第一个业务插件。

- **接入渠道（Channel）可插拔**：企业微信、飞书、钉钉、网页聊天等作为不同渠道接入。
- **核心能力统一**：消息、会话、Agent、知识库、工作流等能力与渠道无关，保持复用。
- **业务插件可扩展**：后续可叠加菜单设计助手、CRM 客户助手、售后助手、知识库问答助手、招标信息助手等。

> 命名寓意：Muse 取"灵感缪斯"之意，易记、国际化，未来扩展为平台也不违和；缩写 Menu Understanding & Service Engine 可解释亦可不强行解释。

> **接入定位**：AI 助手是**给公司内部客服使用的机器人**，不直接对话外部客户。客服把与客户的聊天记录转发给机器人，由 MUSE 整理。这样正好匹配企业微信"智能机器人只能接收内部成员消息"的限制（详见 §四）。

> ⚠️ **愿景 vs 落地**：本节与 §十二 是**长期北极星**；真正的 Phase 1 是一个低吞吐、人工把关的内部工具，实现边界见 **§五**。别在验证业务前先背平台级基建。

---

## 二、系统目标

**要解决的现状问题**——当前餐饮设计公司的链路是：

```
客户 → 微信聊天 → 客服 → 大量聊天记录 → 人工整理 → Excel → 设计师
```

痛点：

- 信息遗漏
- 来回确认
- 重复沟通
- 无法形成知识库
- 无法统计客户需求

**AI 目标链路**：

```
聊天 → 客服转发 → 人工把关 → AI 整理 → 人工审查 → 菜单需求入库 → 报价 → 设计 → 确认
```

```text

        多选客户聊天记录 → 逐条转发 / 合并转发
                       │
          ┌────────────┴─────────────┐
          ▼                          ▼
   智能机器人长连接/webhook       会话存档 SDK
   （Phase 1·逐条转发）         （后续·合并转发 chatrecord）

                       ▼
┌─────────────────────────────────────────────┐

                       │
                       ▼
             业务系统「待整理收件箱」
                       │
          ┌────────────┴─────────────┐
          │   人工把关（选客户 / 筛查）│
          └────────────┬─────────────┘
                       ▼
              LLM 整理（结构化提取）
                       │
                       ▼
            菜单需求（结构化草稿·存版本）
                       │
          ┌────────────┴─────────────┐
          │   人工审查（修正 AI 结果）│
          └────────────┬─────────────┘
                       ▼
┌─────────────────────────────────────────────┐
│                 PostgreSQL                    │
│  Customers · InboxMessages · CurationBatches  │

                       │
                       ▼
                  Next.js Admin
```

> ☝️ 上图是**目标形态**（双 Channel、事件解耦、多 Agent）。**Phase 1 的最小实现**——单 Channel、单人工关口、砍掉重型组件——见 **§五**。

### 为什么强调"接入层绝不直接调用 AI"

无论 Phase 1 用 BackgroundTask 还是后续用事件总线，都要守住这条边界：**网关只负责接收、下载、标准化、秒回 ack；LLM 一律异步。** 好处是消息不丢、可重放、可多消费、网关不被 LLM 拖慢。

## 四、企业微信接入层（关键决策与约束）

### 双 Channel：下游统一，接入方式可换

客服把客户聊天转发给机器人，有两种转发方式，对应两条接入通道。**两者都归一到同一内部消息模型**，下游（收件箱 / LLM / 审查 / 入库）完全复用。

| 渠道 | 客服操作 | 能否解析 | 保留"谁说的" | 原始文件 | 需会话存档 | 阶段 |
| --- | --- | --- | --- | --- | --- | --- |
| 智能机器人长连接 / webhook | 多选 →**逐条转发** | ✅ 逐条 | ❌（靠人工补） | ✅ | 否 | **Phase 1** |
| 会话存档 | **合并转发** | ✅ 解析 `chatrecord` | ✅ | ✅ | 是 | 后续 |

- **Phase 1 先做逐条转发**：不必等会话存档审核即可上线；丢失的"谁说的"由人工把关补齐。
- **会话存档后续接入**：开通后新增一个 Channel，把合并转发的 `chatrecord` 拆成同样的内部消息，灌进**同一个收件箱**——UI、LLM、审查、入库一行不用改。

> WebSocket 与 webhook 仅是**传输差异**（长连接免公网 IP/域名；webhook 需公网 HTTPS 回调 + Token + AESKey），**入站消息类型完全相同**，两者都**不支持合并转发聊天记录**。所以合并转发只能走会话存档。

### 智能机器人可接收的消息类型

`text` · `image` · `mixed`（图文混排）· `voice`（自动转文本）· `file` · `video`。**没有 `chatrecord`（合并转发聊天记录）。**

### 关键约束

- **5 秒内 ack**：收到消息必须 5 秒内应答 → 网关只做"接收 / 下载 / 落库 / ack"，LLM 异步算完再 `aibot_send_msg` 推送。
- **媒体 URL 仅 5 分钟有效**：`image`/`file`/`video` 带 `url` + `aeskey`（AES-256-CBC），网关**立即下载**存储。
- **回复窗口 24 小时**、限速 **30 条/分、1000 条/时**（内部量级足够）。
- **去重**：按 `msgid` 去重（客服可能重复转发同一条）。
- **分组不自动化**：逐条转发是多条独立回调，不做复杂自动分组——统一落入收件箱，由**人工在 UI 指认客户/单据**。

### 会话存档：费用与试用（后续启用）

- **版本**：选**服务版 900 元/人/年**（存文字/图片/文件，含 Excel 原件）；无需音视频，不买企业版（1800）。
- **计费口径**：按**被存档的客服人数**，非客户数、非消息数；**5 人起购** → 最低 4500 元/年。
- **试用**：**30 天免费**，前提企业已认证（✅ 已完成）。后台路径：管理工具 → 会话内容存档 → 申请 30 天试用。
- **SDK**：`WeWorkFinanceSdk`（C 版官方，Python 走 ctypes / 社区封装），按 `seq` 游标轮询拉取、RSA + AES 解密、`sdkfileid` 下载媒体（非实时推送，有秒级延迟，可接受）。
- **上线前 PoC**：合并转发一段真实对话 → 确认能解出 `chatrecord` 嵌套内容 + 用 `sdkfileid` 下到 Excel 原件，再正式付费。

## 五、Phase 1 最小实现（YAGNI 边界）

> 本系统真实形态是**低吞吐、人工把关的内部工具**——客服转发 → 人工把关 → LLM 起草 → 人工确认 → 入库，人是限速器（一天几十到一两百单）。按这个尺子，很多"平台级"组件 Phase 1 用不上。**平台愿景（§一、§十二）保留为北极星，本节只划清"现在做 vs 以后加"。**

### 1. 组件取舍

| 组件 | Phase 1 | 结论与原因 |
| --- | --- | --- |
| **LangGraph** | ❌ 推迟 Phase 3 | 提取就是一次带 Pydantic 结构化输出的 LLM 调用，不需要编排图；真有多 Agent 回环再上 |
| **PaddleOCR** | ❌ 砍 | 菜单图交多模态 LLM 直读，比 OCR→再理解更准；重依赖不背，VLM 扛不住再回头 |
| **Redis Streams** | ❌ 推迟 | 5 秒 ack 用 FastAPI BackgroundTask 即可；批次→LLM 是人工点提交触发。**把网关出口抽象成接口**，量上来再换 |
| **独立 Session / Media Worker** | ❌ 砍 | 不拆微服务，单 FastAPI 应用；会话/批次就是 DB 行 |
| **Qdrant / RAG** | ❌ Phase 5 | 模板推荐早期 Postgres + 关键词够 |
| **MinIO** | ⚠️ 可选 | 保留即可；也可先用本地卷，抽象成存储接口 |

### 2. 收敛为「单人工关口」

原设计两道人工关口（筛查①、审查②）先合并成一道，降低工时、让 ROI 清晰：

```
客服 逐条转发 → 智能机器人（FastAPI 端点，5 秒 ack）
        │
        ▼
BackgroundTask：下载媒体 → 存储 → 写 InboxMessage
        │
        ▼
收件箱：选客户 + 一键提交          ← 筛查退化为"选客户+提交"
        │
        ▼
一次多模态 LLM 调用（Pydantic 结构化）→ MenuRequirement 草稿
        │
        ▼
审查（唯一人工关口）→ 写 PostgreSQL
```

> 若 LLM 直接吃原始转发、准确率不够，**再启用逐条细筛**作为前置关口。先简单，按数据升级，别一上来就上两道关。

### 3. Phase 1 数据模型（4 张表）

`Customer` · `InboxMessage` · `CurationBatch` · `MenuRequirement`（带版本）。其余实体待对应阶段再建（见 §八）。

### 4. 必须处理的边界（否则会坑）

| 风险 | 补法 |
| --- | --- |
| 🔴 分不清"谁说的" | SOP 规定**客服只转客户消息**；或收件箱给每条一个"客户/客服"标注 |
| 🔴 媒体 URL 仅 5 分钟 | 下载**窗口内重试**；最终失败置**错误态**，收件箱标红"文件缺失，请重发" |
| 🟡 逐条转发丢原始时间/顺序 | 按到达 `seq` 保序，收件箱允许**人工重排** |
| 🟡 一批属于哪个客户未知 | 客服转发前先发"客户：张三/单号"，或**提交前强制选客户** |
| 🟢 客服无回执 | 整理完机器人回一条"草稿已生成，点此审查 →链接"（可选） |

### 5. Phase 1 最小栈

`FastAPI + PostgreSQL + 本地卷/MinIO + 一个 OpenAI 兼容多模态模型 + Next.js`，异步用 **FastAPI BackgroundTasks** 顶上，**三个容器起步**。不引入 Redis Streams / LangGraph / PaddleOCR / 微服务拆分。

## 六、消息处理流程（目标形态）

> Phase 1 按 §五.2 的单关口简化执行；本节描述接入完整后的目标流程。

### 1. 网关标准化（关键边界）

企业微信消息经网关统一转换为**内部消息模型**，下游不感知原始协议与渠道。以后接入飞书 / 钉钉 / 会话存档，只需新增网关，其余不动。二进制内容存入存储，事件只携带 `object_key`。

标准化事件示例：

```json
{
  "batch_id": "inbox_xxx",
  "channel": "wecom_bot",
  "forwarded_by": "kf_zhang",
  "timestamp": 1750000000,
  "messages": [
    { "msgid": "m1", "seq": 1, "type": "text",  "content": "四个人" },
    { "msgid": "m2", "seq": 2, "type": "text",  "content": "不要牛肉" },
    { "msgid": "m3", "seq": 3, "type": "image", "object_key": "store://chat/xxx.jpg" },
    { "msgid": "m4", "seq": 4, "type": "file",  "object_key": "store://chat/menu.xlsx" }
  ]
}
```

> 逐条转发下无法可靠区分消息来自客户还是客服，`sender` 归属留待人工把关阶段标注；会话存档接入后 `chatrecord` 自带发送者身份，可自动填充。

### 2. 待整理收件箱 + 人工把关

逐条转发的消息落入**待整理收件箱**。业务系统展示文本 / 图片 / 文件（经存储预览），人工完成：

- **选客户 / 单据**（解决自动分组）——Phase 1 的必做动作
- （可选）去掉不相干消息、标注客户/客服——准确率不足时才启用
- 提交为一个 **CurationBatch** 交给 LLM

### 3. LLM 整理

进入 LLM：图片 → 多模态直读，Excel / PDF → 文档解析，文本直入 → 结构化提取 **MenuRequirement 草稿**。

### 4. 人工审查 → 入库

UI 展示 AI 草稿，人工修正后写入 PostgreSQL；**所有 AI 输出保存版本**，便于追踪与回溯。

> **触发**：以"人工提交 CurationBatch"为主。会话存档接入后可加"合并转发即触发"。

## 七、AI Agent 分层（Phase 3+ 目标）

> **Phase 1 只用"菜单提取"一步**（含图片直读 / 文档解析子步骤），一次结构化 LLM 调用即可，不引入 LangGraph。下表多数 Agent 属 Phase 3 之后，届时用 LangGraph 编排。

| Agent | 职责 | 阶段 |
| --- | --- | --- |
| Menu Extraction Agent | 提取人数、预算、忌口、口味等结构化信息 | Phase 1 |
| Document Agent | 解析 Excel、PDF、Word | Phase 1（并入提取步） |
| Vision Agent | 图片理解 / OCR（菜单图优先 VLM 直读，PaddleOCR 仅兜底） | Phase 1（并入提取步） |
| Conversation Agent | 管理上下文、判断会话阶段 | Phase 3 |
| Quotation Agent | 根据菜单生成报价建议 | Phase 4 |
| Review Agent | 检查信息完整性、提示缺失项 | Phase 3 |

> 所有 AI 输出都应**保存版本**，便于追踪和人工修正。

## 八、数据模型（核心实体）

> **Phase 1 只需 ★ 标的 4 张核心表**；其余待对应阶段再建，别提前铺。

- ★ **Customer**（客户）
- ★ **InboxMessage**（待整理消息：原始转发消息，关联存储 `object_key`，`msgid` 去重，`seq` 保序）
- ★ **CurationBatch**（一次提交给 LLM 的消息集合，关联客户 / 单据）
- ★ **MenuRequirement**（菜单需求，**带版本**）
- Conversation（会话）—— 会话存档接入后才有意义
- Message / Attachment —— 与 InboxMessage 重叠，Phase 1 不单建
- Quote（报价）—— Phase 4
- Order（订单）—— Phase 4
- Knowledge（知识库）—— Phase 5
- PromptTemplate（提示词模板）—— 随 Prompt 管理引入

## 九、后台管理

采用 **Next.js** 自研（而非低代码平台，因业务会逐渐复杂）。

**Phase 1 必做页面：**

- **待整理收件箱**（选客户 + 一键提交；可选逐条筛查/重排）
- **菜单需求审查**（唯一人工关口，修正后入库）

**后续页面：** 客户/会话列表 · 聊天详情 · 菜单编辑 · 报价管理 · 文件管理 · Agent 日志 · Prompt 管理 · 知识库管理 · 系统监控

## 十一、开发阶段规划

| 阶段 | 目标 | 内容 |
| --- | --- | --- |
| **Phase 1** | 端到端最小闭环（逐条转发 + 单人工关口） | 智能机器人接入（逐条转发）、网关标准化 + 媒体下载、待整理收件箱（选客户+提交）、单次 LLM 菜单提取、**人工审查**、入库 PostgreSQL / MinIO。栈：FastAPI + PG + 存储 + 多模态 LLM + Next.js |
| Phase 2 | 平台夯实 + 会话存档 | 接入**会话存档 Channel（合并转发 `chatrecord`）**；按量引入 Redis Streams、会话 / 批次管理 |

| Phase 4 | 业务闭环 | 菜单确认、报价、海报生成、订单管理 |
| Phase 5 | 知识沉淀 | Qdrant、RAG、客户画像、菜单模板推荐 |

## 十二、长期愿景

将 MUSE 演进为统一的 AI 会话处理平台，"菜单整理"仅是第一个业务插件。未来可扩展：

- 菜单设计助手
- 论文服务助手
- 招标信息助手
- CRM 客户助手
- 售后助手
- 知识库问答助手

各聊天平台（企业微信、飞书、钉钉、网页）作为**接入渠道**，而消息、会话、Agent、知识库、工作流等能力保持统一——更具扩展性，也更符合长期产品演进。
